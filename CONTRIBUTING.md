# Contributing to blend-rescript

Thanks for helping keep the ReScript bindings for [`@juspay/blend-design-system`](https://github.com/juspay/blend-design-system) in sync. This guide covers the moving parts you'll need to know.

## Setup

```bash
nvm use              # uses .nvmrc → Node 20
npm install
npm run build        # rescript compile; should be green before any commit
```

## Generating bindings

Bindings are generated by an LLM from the upstream TypeScript declarations in `node_modules/@juspay/blend-design-system/dist/components/*/*.d.ts`. The script `scripts/generate-bindings.mjs` drives the loop.

### One-time: `.env`

Create a `.env` at the repo root:

```
LITELLM_BASE_URL=<your litellm base url>
LITELLM_API_KEY=<your api key>
TEST_KEY=<your key>
```

These are never committed (`.env` is gitignored) and are only used locally or via the CI sync workflow (where they live as repo secrets of the same names).

### Generating one component

```bash
npm run generate -- --component Button
```

### Regenerating everything

```bash
npm run generate -- --all
```

### Retry loop & `.failed.res`

The generator tries up to 3 times per component, feeding ReScript compiler errors back into the next prompt. If all 3 retries fail, the generated file is renamed to `<Component>.failed.res` so it doesn't break compilation of the rest of the tree. Review these by hand, fix or delete, and re-run.

Encountered errors are appended to `scripts/known-errors.md` for the next run's system prompt.

## Syncing with upstream

Stable releases of `@juspay/blend-design-system` drive sync. The CI workflow `sync-bindings.yml`:

- Only runs manually via `Actions → Sync bindings with upstream blend → Run workflow`.
- Refuses prerelease versions (`-beta`, `-alpha`, `-rc`) unless you tick `allow_prerelease`.
- Opens a PR with the regenerated bindings for review.

**Rule:** if you tick `allow_prerelease` to test a `pkg.pr.new` URL or beta, **do not merge** the resulting PR. Use it only for local validation against an unmerged blend change.

## Release flow

1. Update `package.json` version, merge to `main`.
2. Cut a GitHub Release with tag `v<version>` — this triggers `publish-npm.yml` which verifies the tag matches `package.json` and publishes to NPM.
3. `pkg-pr-new.yml` also publishes a preview on every PR and push to `main`, so consumers can test unreleased changes via `https://pkg.pr.new/<owner>/<repo>@<sha>`.

## Checklist before opening a PR

- [ ] `npm run build` green
- [ ] No `src/*.failed.res` files
- [ ] If bindings changed, reviewed diff against upstream
- [ ] Version bump decision noted (`patch` / `minor` / `major`)
- [ ] `CHANGELOG.md` entry for user-facing changes
